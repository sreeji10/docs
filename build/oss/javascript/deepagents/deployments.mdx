This page covers how to take your local Deep Agent to production.

Deep Agents deploy on [LangSmith Deployment](/langsmith/deployment) (LSD), which handles infrastructure for you — [assistants](/langsmith/assistants), [threads](/langsmith/use-threads), [runs](/langsmith/runs), a provisioned store and checkpointer — and can expose your agent via [MCP](/langsmith/server-mcp), [A2A](/langsmith/server-a2a), [webhooks](/langsmith/use-webhooks), or [cron](/langsmith/cron-jobs). For general deployment setup, see the [LangSmith Deployment quickstart](/langsmith/deployment-quickstart).

## Overview

When deploying a Deep Agent, there are a few key decisions to make:

- **[Memory](#memory)** — How to scope persistent data (preferences, instructions, [skills](/oss/javascript/deepagents/skills)) that carries across conversations
- **[Files](#files)** — How to configure the agent's filesystem — which [backend](/oss/javascript/deepagents/backends) to use, what's ephemeral vs persistent
- **[Sandboxes](#sandboxes)** — How to manage isolated environments for code execution
- **[Resilience middleware](#production-middleware)** — How to handle transient failures like rate limits and provider outages

## Memory

A deployed agent needs to remember things across conversations — user preferences, learned instructions, project context. Without persistent memory, every conversation starts from scratch. This persistent storage is also where [skills](/oss/javascript/deepagents/skills) (SKILL.md) and agent memories (AGENTS.md) are loaded from.

Use a [`CompositeBackend`](https://reference.langchain.com/javascript/deepagents/backends/CompositeBackend) to give the agent both ephemeral scratch space and persistent [long-term memory](/oss/javascript/deepagents/long-term-memory). Anything the agent writes to `/memories/` persists in the LangGraph [Store](/langsmith/custom-store) that LSD provisions automatically. Everything else is ephemeral and scoped to the current conversation.

The [namespace factory](/oss/javascript/deepagents/backends#namespace-factories) controls who can access that persistent memory — whether it's private to a single user or shared across an [assistant](/langsmith/assistants):

<Tabs>
  <Tab title="Per-user (most common)">
    Each user gets private persistent memory across all their conversations. Use this for personal preferences, instructions, and learned context.



```typescript
import { createDeepAgent, CompositeBackend, StateBackend, StoreBackend } from "deepagents";

const agent = createDeepAgent({
  backend: (rt) => new CompositeBackend(
    new StateBackend(rt),  // Ephemeral scratch space
    {
      "/memories/": new StoreBackend(rt, {
        namespace: (ctx) => [ctx.runtime.context.userId],
      }),
    },
  ),
});
```


  </Tab>
  <Tab title="Per-assistant">
    All users of the same [assistant](/langsmith/assistants) share memory. Use this for shared team knowledge or project documentation.



```typescript
import { getConfig } from "@langchain/langgraph";
import { createDeepAgent, CompositeBackend, StateBackend, StoreBackend } from "deepagents";

const agent = createDeepAgent({
  backend: (rt) => new CompositeBackend(
    new StateBackend(rt),
    {
      "/memories/": new StoreBackend(rt, {
        namespace: (ctx) => {
          const config = getConfig();
          return [config.metadata.assistantId];
        },
      }),
    },
  ),
});
```


  </Tab>
  <Tab title="Per-user per-thread">
    Memory is private to a specific user *and* a specific conversation. Use this when the agent's persistent state should be conversation-specific — for example, a project workspace that each thread manages independently.



```typescript
import { getConfig } from "@langchain/langgraph";
import { createDeepAgent, CompositeBackend, StateBackend, StoreBackend } from "deepagents";

const agent = createDeepAgent({
  backend: (rt) => new CompositeBackend(
    new StateBackend(rt),
    {
      "/memories/": new StoreBackend(rt, {
        namespace: (ctx) => {
          const config = getConfig();
          return [ctx.runtime.context.userId, config.configurable.threadId];
        },
      }),
    },
  ),
});
```


  </Tab>
</Tabs>

You can also read and write to the store from your application code using the [Store API](/langsmith/custom-store). See [accessing memories from external code](/oss/javascript/deepagents/long-term-memory#accessing-memories-from-external-code) for examples.

For the full namespace factory API, see [namespace factories](/oss/javascript/deepagents/backends#namespace-factories). For memory patterns like self-improving instructions and knowledge bases, see [long-term memory](/oss/javascript/deepagents/long-term-memory).

## Files

Deep Agents expose a virtual filesystem to the agent via tools like `read_file`, `write_file`, `edit_file`, `ls`, `glob`, and `grep`. The [backend](/oss/javascript/deepagents/backends) you choose determines where those files live.

For most deployed agents, the [`CompositeBackend`](https://reference.langchain.com/javascript/deepagents/backends/CompositeBackend) setup from the [memory examples above](#memory) is all you need — [`StateBackend`](https://reference.langchain.com/javascript/deepagents/backends/StateBackend) for ephemeral scratch work, [`StoreBackend`](https://reference.langchain.com/javascript/deepagents/backends/StoreBackend) for anything that should persist. If your agent needs to execute code, swap `StateBackend` for a [sandbox](#sandboxes).

<Warning>
    Do not use `FilesystemBackend` or `LocalShellBackend` in deployed agents. They access the host filesystem directly and are not isolated between users or threads.
</Warning>

<Warning>
    `StateBackend` stores files in LangGraph state, which is checkpointed at every agent step. Avoid writing very large files to state — they will be serialized and persisted on every step, increasing checkpoint size and latency. For large artifacts, use a `StoreBackend` route or a [sandbox](/oss/javascript/deepagents/sandboxes).
</Warning>

For the full list of backends and how to build custom ones, see [backends](/oss/javascript/deepagents/backends).

## Sandboxes

If your agent needs to execute code, it needs an isolated environment. [Sandboxes](/oss/javascript/deepagents/sandboxes) give the agent a filesystem and shell access in a container that can't affect your infrastructure. In production, always use a sandbox instead of `LocalShellBackend`, which has no isolation.

The key decision is how long a sandbox lives — does each conversation get a fresh one, or do conversations share a persistent environment?

| Scope | Sandbox ID stored on | Lifecycle | Best for |
|---|---|---|---|
| **Thread-scoped** | [Thread](/langsmith/use-threads) metadata | Fresh per conversation, cleaned up on TTL | Data analysis, one-off coding |
| **Assistant-scoped** | [Assistant](/langsmith/assistants) config | Shared across all conversations | Persistent dev environments |

<Tabs>
  <Tab title="Thread-scoped (most common)">
    Each conversation gets its own sandbox. The sandbox is created when the first [run](/langsmith/runs) starts and reused for follow-up messages on the same [thread](/langsmith/use-threads). Cleaned up when the thread ends or the sandbox [TTL](/langsmith/configure-ttl) expires. Store the sandbox ID in thread metadata so follow-up runs reconnect.
  </Tab>
  <Tab title="Assistant-scoped">
    All conversations share one sandbox. The sandbox ID is stored on the [assistant](/langsmith/assistants) config, so every conversation returns to the same environment. Files, installed packages, and cloned repositories persist across conversations.

    <Warning>
        Assistant-scoped sandboxes accumulate state over time. Implement periodic cleanup, use snapshots to reset, or configure a TTL to prevent unbounded resource usage.
    </Warning>
  </Tab>
</Tabs>

To combine a sandbox with persistent memory, use the sandbox as the default backend in a `CompositeBackend`:



```typescript
import { createDeepAgent, CompositeBackend, StoreBackend } from "deepagents";

const agent = createDeepAgent({
  backend: (rt) => new CompositeBackend(
    sandboxBackend,  // Code execution + working files
    {
      "/memories/": new StoreBackend(rt, {
        namespace: (ctx) => [ctx.runtime.context.userId],
      }),
    },
  ),
});
```


For provider setup, security, and lifecycle patterns, see the full [sandboxes guide](/oss/javascript/deepagents/sandboxes).

## Production middleware

In production, transient failures are common — model API rate limits, provider outages, flaky tool calls. Add [resilience middleware](/oss/javascript/langchain/middleware/built-in) to handle these automatically:



```typescript
import { createDeepAgent } from "deepagents";
import {
  ModelRetryMiddleware,
  ModelFallbackMiddleware,
  ToolRetryMiddleware,
} from "@langchain/langgraph";

const agent = createDeepAgent({
  model: "claude-sonnet-4-6",
  middleware: [
    new ModelRetryMiddleware({ maxRetries: 3 }),  // Retry on transient API failures
    new ModelFallbackMiddleware("gpt-4.1"),       // Fall back if primary model is down
    new ToolRetryMiddleware({ maxRetries: 2 }),   // Retry flaky tool calls
  ],
  ...
});
```


| Middleware | Purpose |
|---|---|
| [`ModelRetryMiddleware`](https://reference.langchain.com/javascript/langchain/index/modelRetryMiddleware) | Retry on rate limits, timeouts, server errors |
| [`ModelFallbackMiddleware`](https://reference.langchain.com/javascript/langchain/index/modelFallbackMiddleware) | Try alternative models on failure |
| [`ToolRetryMiddleware`](https://reference.langchain.com/javascript/langchain/index/toolRetryMiddleware) | Retry flaky tool calls |
| [`ModelCallLimitMiddleware`](https://reference.langchain.com/javascript/langchain/index/modelCallLimitMiddleware) | Cap model calls per run |
| [`ToolCallLimitMiddleware`](https://reference.langchain.com/javascript/langchain/index/toolCallLimitMiddleware) | Cap tool executions per run |

For full configuration options (backoff, jitter, exception filtering, failure handling), see [prebuilt middleware](/oss/javascript/langchain/middleware/built-in).
