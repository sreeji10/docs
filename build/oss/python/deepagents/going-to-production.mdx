A local prototype typically runs as a single user in a single conversation. Going to production introduces new challenges: multiple users need isolated state, conversations need clear boundaries between ephemeral scratch work and persistent memory, and your agent needs to handle failures and stay within cost and safety limits.

This page covers:
- **[LangSmith Deployments](#langsmith-deployments)**: managed infrastructure with auth, webhooks, and cron
- **[Memory](#memory)**: persist information across conversations
- **[Execution environment](#execution-environment)**: file storage and code execution
- **[Guardrails](#guardrails)**: flow limits, resilience, and data privacy
- **[Frontend](#frontend)**: connect your UI to a deployed agent

## LangSmith Deployments

The fastest way to get a Deep Agent into production is [LangSmith Deployments](/langsmith/deployment). It provisions the infrastructure your agent needs — [assistants](/langsmith/assistants), [threads](/langsmith/use-threads), [runs](/langsmith/runs), a store, and a checkpointer — so you don't have to set these up yourself. It also gives you [authentication](/langsmith/auth), [webhooks](/langsmith/use-webhooks), [cron jobs](/langsmith/cron-jobs), and [observability](/langsmith/observability) out of the box, and can expose your agent via [MCP](/langsmith/server-mcp) or [A2A](/langsmith/server-a2a).

For setup instructions, see the [LangSmith Deployments quickstart](/langsmith/deployment-quickstart).

## Memory

Without memory, every conversation starts from scratch. Memory lets your agent retain information across conversations — user preferences, learned instructions, past experiences — so it can personalize its behavior over time. For an overview of memory types, see the [memory concepts guide](/oss/python/concepts/memory).

### Scoping

Memories have different scopes depending on how they're shared:

- **Assistant** (most common): all users of the same assistant share memory
- **Assistant × user**: each user gets private memory within a given assistant
- **Global (user)**: memory that follows a user across all assistants
- **Global (org)**: memory shared across the entire org

For the full scoping API, see [namespace factories](/oss/python/deepagents/backends#namespace-factories).

### Configuration

In Deep Agents, memory is stored as files in a virtual filesystem. By default, files only last for a single conversation. To persist them, route a path like `/memories/` to a [`StoreBackend`](https://reference.langchain.com/python/deepagents/backends/store/StoreBackend) that writes to the LangGraph [Store](/langsmith/custom-store). Use a [`CompositeBackend`](https://reference.langchain.com/python/deepagents/backends/composite/CompositeBackend) to give the agent both ephemeral scratch space and persistent [long-term memory](/oss/python/deepagents/long-term-memory).

<Tabs>
  <Tab title="Assistant (most common)">
    Namespace by `assistant_id`. All users of the same assistant read and write to the same memory.

```python
from langgraph.config import get_config
from deepagents import create_deep_agent
from deepagents.backends import CompositeBackend, StateBackend, StoreBackend

agent = create_deep_agent(
    backend=lambda rt: CompositeBackend(
        default=StateBackend(rt),
        routes={
            "/memories/": StoreBackend(
                rt,
                namespace=lambda ctx: (
                    get_config()["metadata"]["assistant_id"],
                ),
            ),
        },
    ),
    system_prompt="""You have persistent memory at /memories/.

    Read /memories/instructions.txt at the start of each conversation for
    accumulated knowledge and preferences. When you learn something that
    should persist, update that file.""",
)
```




  </Tab>
  <Tab title="Assistant × user">
    Namespace by `(assistant_id, user_id)`. Each user gets private memory within a given assistant.

```python
from langgraph.config import get_config
from deepagents import create_deep_agent
from deepagents.backends import CompositeBackend, StateBackend, StoreBackend

agent = create_deep_agent(
    backend=lambda rt: CompositeBackend(
        default=StateBackend(rt),
        routes={
            "/memories/": StoreBackend(
                rt,
                namespace=lambda ctx: (
                    get_config()["metadata"]["assistant_id"],
                    ctx.runtime.context.user_id,
                ),
            ),
        },
    ),
)
```




  </Tab>
  <Tab title="Global">
    Namespace by `user_id` or `org_id`. Memory is shared across all assistants.

```python
from deepagents import create_deep_agent
from deepagents.backends import CompositeBackend, StateBackend, StoreBackend

agent = create_deep_agent(
    backend=lambda rt: CompositeBackend(
        default=StateBackend(rt),
        routes={
            "/memories/": StoreBackend(
                rt,
                # Per-user global: (ctx.runtime.context.user_id,)
                # Per-org global: (ctx.runtime.context.org_id,)
                namespace=lambda ctx: (ctx.runtime.context.user_id,),
            ),
        },
    ),
)
```




  </Tab>
</Tabs>

You can also read and write to the store from your application code using the [Store API](/langsmith/custom-store). See [accessing memories from external code](/oss/python/deepagents/long-term-memory#accessing-memories-from-external-code) for examples.

For the full namespace factory API, see [namespace factories](/oss/python/deepagents/backends#namespace-factories). For memory patterns like self-improving instructions and knowledge bases, see [long-term memory](/oss/python/deepagents/long-term-memory).

## Execution environment

Locally, agents can read and write files on disk and run shell commands directly. In production, you need to think about isolation and persistence.

### Filesystem

`FilesystemBackend` and `LocalShellBackend` access the host directly — don't use them in deployed agents. Choose a backend based on what needs to persist:

- [`StateBackend`](https://reference.langchain.com/python/deepagents/backends/state/StateBackend): ephemeral scratch space, scoped to a single conversation. Checkpointed at every step, so avoid writing large files.
- [`StoreBackend`](https://reference.langchain.com/python/deepagents/backends/store/StoreBackend): persistent storage that survives across conversations. Scope with a [namespace factory](/oss/python/deepagents/backends#namespace-factories).
- [`CompositeBackend`](https://reference.langchain.com/python/deepagents/backends/composite/CompositeBackend): mix both — ephemeral scratch by default with persistent routes for specific paths like `/memories/`.

For the full list of backends and how to build custom ones, see [backends](/oss/python/deepagents/backends).

### Sandboxes

If your agent needs to run code — not just read and write files — use a [sandbox](/oss/python/deepagents/sandboxes). Sandboxes provide both a filesystem and an `execute` tool for running shell commands, all inside an isolated container.

#### Lifecycle

The key decision is how long a sandbox lives. Does each conversation get a fresh one, or do conversations share a persistent environment?

| Scope | Sandbox ID stored on | Lifecycle | Best for |
|---|---|---|---|
| **Thread-scoped** | [Thread](/langsmith/use-threads) metadata | Fresh per conversation, cleaned up on TTL | Data analysis, one-off coding |
| **Assistant-scoped** | [Assistant](/langsmith/assistants) config | Shared across all conversations | Persistent dev environments |

<Tabs>
  <Tab title="Thread-scoped (most common)">
    Each conversation gets its own sandbox. The sandbox is created when the first [run](/langsmith/runs) starts and reused for follow-up messages on the same [thread](/langsmith/use-threads). Cleaned up when the thread ends or the sandbox [TTL](/langsmith/configure-ttl) expires. Store the sandbox ID in thread metadata so follow-up runs reconnect.

```python
import uuid

from daytona import CreateSandboxFromSnapshotParams, Daytona
from langchain_daytona import DaytonaSandbox
from deepagents import create_deep_agent

client = Daytona()
thread_id = str(uuid.uuid4())

# Get or create sandbox by thread_id
try:
    sandbox = client.find_one(labels={"thread_id": thread_id})
except Exception:
    params = CreateSandboxFromSnapshotParams(
        labels={"thread_id": thread_id},
        auto_delete_interval=3600,  # TTL: clean up when idle
    )
    sandbox = client.create(params)

backend = DaytonaSandbox(sandbox=sandbox)
agent = create_deep_agent(backend=backend)

result = agent.invoke(
    {"messages": [{"role": "user", "content": "Run hello world"}]},
    config={"configurable": {"thread_id": thread_id}},
)
```




  </Tab>
  <Tab title="Assistant-scoped">
    All conversations share one sandbox. The sandbox ID is stored on the [assistant](/langsmith/assistants) config, so every conversation returns to the same environment. Files, installed packages, and cloned repositories persist across conversations.

```python
from langgraph.config import get_config
from daytona import CreateSandboxFromSnapshotParams, Daytona
from langchain_daytona import DaytonaSandbox
from deepagents import create_deep_agent

client = Daytona()

def get_or_create_sandbox():
    assistant_id = get_config()["metadata"]["assistant_id"]
    try:
        return client.find_one(labels={"assistant_id": assistant_id})
    except Exception:
        return client.create(
            CreateSandboxFromSnapshotParams(labels={"assistant_id": assistant_id})
        )

sandbox = get_or_create_sandbox()
backend = DaytonaSandbox(sandbox=sandbox)
agent = create_deep_agent(backend=backend)
```




    <Warning>
        Assistant-scoped sandboxes accumulate state over time. Implement periodic cleanup, use snapshots to reset, or configure a TTL to prevent unbounded resource usage.
    </Warning>
  </Tab>
</Tabs>

#### Working with files

Your application code can't directly access files inside the sandbox. Use the file transfer APIs to move data across that boundary:

- **Seed files before the agent runs** with `upload_files()` — for example, copy a [skill](/oss/python/deepagents/skills) script into the sandbox so the agent can execute it
- **Retrieve artifacts after the agent finishes** with `download_files()` — pull out generated reports, plots, or other outputs

For code examples with each sandbox provider, see [working with files](/oss/python/deepagents/sandboxes#working-with-files).

#### Syncing memory

If your agent uses persistent [memory](/oss/python/deepagents/long-term-memory) stored in the LangGraph [Store](/langsmith/custom-store), you can sync it into and out of the sandbox using the file transfer APIs combined with the [Store API](/langsmith/custom-store):

1. **Before the run**: read from the store, then `upload_files()` into the sandbox
2. **After the run**: `download_files()` from the sandbox, then write back to the store

```python
from langgraph_sdk import get_client

client = get_client(url="<DEPLOYMENT_URL>")

# Before the run: seed memories into the sandbox
items = await client.store.search_items((user_id, "memories"))
files = [(item.key, item.value["content"].encode()) for item in items]
sandbox_backend.upload_files(files)

# ... run the agent ...

# After the run: sync updated memories back to the store
results = sandbox_backend.download_files(["memories/instructions.txt"])
for result in results:
    if result.content is not None:
        await client.store.put_item(
            (user_id, "memories"),
            key=result.path,
            value={"content": result.content.decode()},
        )
```




For provider setup, security, and lifecycle patterns, see the full [sandboxes guide](/oss/python/deepagents/sandboxes).

## Guardrails

In production, you want to prevent runaway agents, recover from transient failures, and protect sensitive data. Deep Agents support [middleware](/oss/python/langchain/middleware/built-in) that wraps model and tool calls to handle these concerns.

For a broader view of error handling strategies (transient vs LLM-recoverable vs user-fixable errors), see [Handle errors appropriately](/oss/python/langgraph/thinking-in-langgraph#handle-errors-appropriately).

### Control flow

Prevent runaway agents by capping model and tool call volume:

```python
from langchain.agents.middleware import ModelCallLimitMiddleware, ToolCallLimitMiddleware
from deepagents import create_deep_agent

agent = create_deep_agent(
    model="claude-sonnet-4-6",
    middleware=[
        ModelCallLimitMiddleware(max_calls=50),
        ToolCallLimitMiddleware(max_calls=200),
    ],
)
```




| Middleware | Purpose |
|---|---|
| [`ModelCallLimitMiddleware`](https://reference.langchain.com/python/langchain/agents/middleware/model_call_limit/ModelCallLimitMiddleware) | Cap model calls per run to prevent runaway loops |
| [`ToolCallLimitMiddleware`](https://reference.langchain.com/python/langchain/agents/middleware/tool_call_limit/ToolCallLimitMiddleware) | Cap tool executions per run |

### Resilience

Automatically recover from transient failures like rate limits, provider outages, and flaky tool calls:

```python
from langchain.agents.middleware import (
    ModelRetryMiddleware,
    ModelFallbackMiddleware,
    ToolRetryMiddleware,
)
from deepagents import create_deep_agent

agent = create_deep_agent(
    model="claude-sonnet-4-6",
    middleware=[
        ModelRetryMiddleware(max_retries=3),
        ModelFallbackMiddleware("gpt-4.1"),
        ToolRetryMiddleware(
            max_retries=2,
            tools=["search", "fetch_url"],  # Only retry tools with transient failures
        ),
    ],
)
```




| Middleware | Purpose |
|---|---|
| [`ModelRetryMiddleware`](https://reference.langchain.com/python/langchain/agents/middleware/model_retry/ModelRetryMiddleware) | Retry on rate limits, timeouts, server errors |
| [`ModelFallbackMiddleware`](https://reference.langchain.com/python/langchain/agents/middleware/model_fallback/ModelFallbackMiddleware) | Try alternative models on failure |
| [`ToolRetryMiddleware`](https://reference.langchain.com/python/langchain/agents/middleware/tool_retry/ToolRetryMiddleware) | Retry failed tool calls, optionally scoped to specific tools |

### Data privacy

Detect and handle sensitive data before it reaches the model or gets stored in logs:

```python
from langchain.agents.middleware import PIIMiddleware
from deepagents import create_deep_agent

agent = create_deep_agent(
    model="claude-sonnet-4-6",
    middleware=[
        PIIMiddleware("email", strategy="redact", apply_to_input=True),
        PIIMiddleware("credit_card", strategy="mask", apply_to_input=True),
    ],
)
```




| Middleware | Purpose |
|---|---|
| [`PIIMiddleware`](https://reference.langchain.com/python/langchain/agents/middleware/pii/PIIMiddleware) | Detect and redact or mask PII (emails, credit cards, custom patterns) |

For full configuration options (backoff, jitter, exception filtering, PII strategies, custom detectors), see [prebuilt middleware](/oss/python/langchain/middleware/built-in).

## Frontend

Deep Agents use [`useStream`](/oss/python/langchain/frontend/overview) to connect your UI to the agent backend. `useStream` is a frontend hook (available for React, Vue, Svelte, and Angular) that streams messages, subagent progress, and custom state from your agent in real time.

Locally, `useStream` points at `http://localhost:2024`. In production, point it at your [LangSmith Deployment](/langsmith/deployment) and configure reconnection so users don't lose progress if their connection drops.

```tsx
import { useStream } from "@langchain/react";

function App() {
  const stream = useStream<typeof agent>({
    apiUrl: "https://your-deployment.langsmith.dev",
    assistantId: "agent",
    reconnectOnMount: true,    // Resume stream after page refresh or navigation
    fetchStateHistory: true,   // Load full thread history on mount
  });
}
```

`reconnectOnMount` picks up an in-progress run automatically — if a user refreshes while the agent is working, they'll see it continue rather than a blank screen. `fetchStateHistory` loads the full conversation history for the thread, so returning users see previous messages.

For deep agent workflows that spawn many subagents, set a high `recursionLimit` when submitting to avoid cutting off long-running executions:

```tsx
stream.submit(
  { messages: [{ type: "human", content: text }] },
  {
    streamSubgraphs: true,
    config: { recursionLimit: 10000 },
  },
);
```

For UI patterns specific to deep agents — subagent cards, todo lists, and custom state rendering — see the [frontend guide](/oss/python/deepagents/frontend/overview).
