title: Going to production

A local prototype typically runs as a single user in a single conversation. Going to production introduces new challenges: multiple users need isolated state, conversations need clear boundaries between ephemeral scratch work and persistent memory, and your agent needs to handle failures and stay within cost and safety limits.

This page covers:
- **[LangSmith Deployments](#langsmith-deployments)**: managed infrastructure with auth, webhooks, cron, and observability
- **[Memory](#memory)**: persist information across conversations, scoped by assistant, user, or org
- **[Execution environment](#execution-environment)**: choose where files live and whether the agent can run code
- **[Guardrails](#guardrails)**: control flow limits, resilience, and data privacy

## LangSmith Deployments

[LangSmith Deployments](/langsmith/deployment) handles production infrastructure for you, including [assistants](/langsmith/assistants), [threads](/langsmith/use-threads), [runs](/langsmith/runs), a provisioned store, and a checkpointer. It also provides [authentication](/langsmith/auth), [webhooks](/langsmith/use-webhooks), [cron jobs](/langsmith/cron-jobs), [observability](/langsmith/observability), and can expose your agent via [MCP](/langsmith/server-mcp) or [A2A](/langsmith/server-a2a).

For setup instructions, see the [LangSmith Deployments quickstart](/langsmith/deployment-quickstart).

## Memory

Memory is inherently cross-thread — it persists information across conversations. Without it, every conversation starts from scratch. The [namespace factory](/oss/python/deepagents/backends#namespace-factories) controls who shares memory:

- **Assistant × user** (most common): each user gets private memory within a given assistant. Personal preferences, instructions, learned context.
- **Assistant**: all users of the same assistant share memory. Shared team knowledge, project documentation.
- **Global**: memory not tied to any assistant. Scope by user for cross-assistant preferences, or by org for company-wide knowledge.

Use a [`CompositeBackend`](https://reference.langchain.com/python/deepagents/backends/composite/CompositeBackend) to give the agent both ephemeral scratch space and persistent [long-term memory](/oss/python/deepagents/long-term-memory). Anything the agent writes to `/memories/` persists in the LangGraph [Store](/langsmith/custom-store). Everything else is ephemeral and scoped to the current conversation.

<Tabs>
  <Tab title="Assistant × user (most common)">
    Each user gets private memory within a given assistant. Personal preferences, instructions, learned context.

```python
from langgraph.config import get_config
from deepagents import create_deep_agent
from deepagents.backends import CompositeBackend, StateBackend, StoreBackend

agent = create_deep_agent(
    backend=lambda rt: CompositeBackend(
        default=StateBackend(rt),
        routes={
            "/memories/": StoreBackend(
                rt,
                namespace=lambda ctx: (
                    get_config()["metadata"]["assistant_id"],
                    ctx.runtime.context.user_id,
                ),
            ),
        },
    ),
)
```




  </Tab>
  <Tab title="Assistant">
    All users of the same [assistant](/langsmith/assistants) share memory. Shared team knowledge, project documentation.

```python
from langgraph.config import get_config
from deepagents import create_deep_agent
from deepagents.backends import CompositeBackend, StateBackend, StoreBackend

agent = create_deep_agent(
    backend=lambda rt: CompositeBackend(
        default=StateBackend(rt),
        routes={
            "/memories/": StoreBackend(
                rt,
                namespace=lambda ctx: (
                    get_config()["metadata"]["assistant_id"],
                ),
            ),
        },
    ),
)
```




  </Tab>
  <Tab title="Global">
    Memory not tied to any assistant. Scope by `user_id` for cross-assistant preferences, or by `org_id` for company-wide knowledge.

```python
from deepagents import create_deep_agent
from deepagents.backends import CompositeBackend, StateBackend, StoreBackend

agent = create_deep_agent(
    backend=lambda rt: CompositeBackend(
        default=StateBackend(rt),
        routes={
            "/memories/": StoreBackend(
                rt,
                # Per-user global: (ctx.runtime.context.user_id,)
                # Per-org global: (ctx.runtime.context.org_id,)
                namespace=lambda ctx: (ctx.runtime.context.user_id,),
            ),
        },
    ),
)
```




  </Tab>
</Tabs>

You can also read and write to the store from your application code using the [Store API](/langsmith/custom-store). See [accessing memories from external code](/oss/python/deepagents/long-term-memory#accessing-memories-from-external-code) for examples.

For the full namespace factory API, see [namespace factories](/oss/python/deepagents/backends#namespace-factories). For memory patterns like self-improving instructions and knowledge bases, see [long-term memory](/oss/python/deepagents/long-term-memory).

## Execution environment

Deep Agents expose a virtual filesystem to the agent via tools like `read_file`, `write_file`, `edit_file`, `ls`, `glob`, and `grep`. The [backend](/oss/python/deepagents/backends) you choose determines where those files live and whether the agent can execute code. Like memory, the execution environment needs scoping in production.

### Filesystem

For agents that don't need to execute code, choose a filesystem backend based on your persistence needs:

| Backend | Persistence | Best for |
|---|---|---|
| [`StateBackend`](https://reference.langchain.com/python/deepagents/backends/state/StateBackend) | Ephemeral (single thread) | Scratch work, intermediate results |
| [`StoreBackend`](https://reference.langchain.com/python/deepagents/backends/store/StoreBackend) | Persistent (cross-thread) | Durable file storage scoped by namespace |
| [`CompositeBackend`](https://reference.langchain.com/python/deepagents/backends/composite/CompositeBackend) | Mixed | Ephemeral scratch + persistent routes (e.g., `/memories/`) |

<Warning>
    Do not use `FilesystemBackend` or `LocalShellBackend` in deployed agents. They access the host filesystem directly and are not isolated between users or threads.
</Warning>

<Warning>
    `StateBackend` stores files in LangGraph state, which is checkpointed at every agent step. Avoid writing very large files to state. They will be serialized and persisted on every step, increasing checkpoint size and latency. For large artifacts, use a `StoreBackend` route or a [sandbox](#sandboxes).
</Warning>

For the full list of backends and how to build custom ones, see [backends](/oss/python/deepagents/backends).

### Sandboxes

If your agent needs to execute code, it needs an isolated environment. [Sandboxes](/oss/python/deepagents/sandboxes) give the agent a filesystem and shell access in a container that can't affect your infrastructure. In production, always use a sandbox instead of `LocalShellBackend`, which has no isolation.

#### Lifecycle

The key decision is how long a sandbox lives. Does each conversation get a fresh one, or do conversations share a persistent environment?

| Scope | Sandbox ID stored on | Lifecycle | Best for |
|---|---|---|---|
| **Thread-scoped** | [Thread](/langsmith/use-threads) metadata | Fresh per conversation, cleaned up on TTL | Data analysis, one-off coding |
| **Assistant-scoped** | [Assistant](/langsmith/assistants) config | Shared across all conversations | Persistent dev environments |

<Tabs>
  <Tab title="Thread-scoped (most common)">
    Each conversation gets its own sandbox. The sandbox is created when the first [run](/langsmith/runs) starts and reused for follow-up messages on the same [thread](/langsmith/use-threads). Cleaned up when the thread ends or the sandbox [TTL](/langsmith/configure-ttl) expires. Store the sandbox ID in thread metadata so follow-up runs reconnect.
  </Tab>
  <Tab title="Assistant-scoped">
    All conversations share one sandbox. The sandbox ID is stored on the [assistant](/langsmith/assistants) config, so every conversation returns to the same environment. Files, installed packages, and cloned repositories persist across conversations.

    <Warning>
        Assistant-scoped sandboxes accumulate state over time. Implement periodic cleanup, use snapshots to reset, or configure a TTL to prevent unbounded resource usage.
    </Warning>
  </Tab>
</Tabs>

#### Combine with memory

When you use a `CompositeBackend` with a sandbox, the sandbox's `execute` tool only sees files in the sandbox's own filesystem — not files stored in persistent memory via `StoreBackend`. For example, a user-scoped [skill](/oss/python/deepagents/skills) script stored in the store can be *read* by the agent but not directly *executed*.

Use a `CompositeBackend` with the sandbox as the default backend to give the agent both code execution and persistent storage:

```python
from deepagents import create_deep_agent
from deepagents.backends import CompositeBackend, StoreBackend

agent = create_deep_agent(
    backend=lambda rt: CompositeBackend(
        default=sandbox_backend,  # Code execution + working files
        routes={
            "/memories/": StoreBackend(
                rt,
                namespace=lambda ctx: (ctx.runtime.context.user_id,),
            ),
        },
    ),
)
```




To make persistent files executable in the sandbox, there are two approaches:

<Tabs>
  <Tab title="Seed at startup">
    Before the agent runs, read files from the [Store API](/langsmith/custom-store) and upload them into the sandbox using [`upload_files()`](/oss/python/deepagents/sandboxes#seeding-the-sandbox). This ensures files are in the sandbox filesystem and ready to execute from the first turn.

```python
from langgraph_sdk import get_client

client = get_client(url="<DEPLOYMENT_URL>")

# Read user-scoped skill scripts from the store
items = await client.store.search_items(
    (user_id, "filesystem"),
    filter={"prefix": "/skills/"},
)

# Upload them into the sandbox so the agent can execute them
files = [(item.key, "\n".join(item.value["content"]).encode()) for item in items]
sandbox_backend.upload_files(files)
```




  </Tab>
  <Tab title="Agent copies on demand">
    With a `CompositeBackend`, the agent can read a file from the store route (e.g., `/memories/scripts/analyze.py`), write it to the sandbox workspace (e.g., `/workspace/analyze.py`), and then execute it. This happens naturally if the skill instructions tell the agent to copy files before running them. No application-level orchestration is needed, but there is an extra read-write round trip per file.
  </Tab>
</Tabs>

For provider setup, security, and lifecycle patterns, see the full [sandboxes guide](/oss/python/deepagents/sandboxes).

## Guardrails

Guardrails wrap model and tool calls to add resilience, enforce limits, and protect data. These use [middleware](/oss/python/langchain/middleware/built-in) and apply to any LangChain agent, not just Deep Agents.

For a broader view of error handling strategies (transient vs LLM-recoverable vs user-fixable errors), see [Handle errors appropriately](/oss/python/langgraph/thinking-in-langgraph#handle-errors-appropriately).

### Control flow

Prevent runaway agents by capping model and tool call volume:

```python
from langchain.agents.middleware import ModelCallLimitMiddleware, ToolCallLimitMiddleware
from deepagents import create_deep_agent

agent = create_deep_agent(
    model="claude-sonnet-4-6",
    middleware=[
        ModelCallLimitMiddleware(max_calls=50),
        ToolCallLimitMiddleware(max_calls=200),
    ],
)
```




| Middleware | Purpose |
|---|---|
| [`ModelCallLimitMiddleware`](https://reference.langchain.com/python/langchain/agents/middleware/model_call_limit/ModelCallLimitMiddleware) | Cap model calls per run to prevent runaway loops |
| [`ToolCallLimitMiddleware`](https://reference.langchain.com/python/langchain/agents/middleware/tool_call_limit/ToolCallLimitMiddleware) | Cap tool executions per run |

### Resilience

Automatically recover from transient failures like rate limits, provider outages, and flaky tool calls:

```python
from langchain.agents.middleware import (
    ModelRetryMiddleware,
    ModelFallbackMiddleware,
    ToolRetryMiddleware,
)
from deepagents import create_deep_agent

agent = create_deep_agent(
    model="claude-sonnet-4-6",
    middleware=[
        ModelRetryMiddleware(max_retries=3),
        ModelFallbackMiddleware("gpt-4.1"),
        ToolRetryMiddleware(
            max_retries=2,
            tools=["search", "fetch_url"],  # Only retry tools with transient failures
        ),
    ],
)
```




| Middleware | Purpose |
|---|---|
| [`ModelRetryMiddleware`](https://reference.langchain.com/python/langchain/agents/middleware/model_retry/ModelRetryMiddleware) | Retry on rate limits, timeouts, server errors |
| [`ModelFallbackMiddleware`](https://reference.langchain.com/python/langchain/agents/middleware/model_fallback/ModelFallbackMiddleware) | Try alternative models on failure |
| [`ToolRetryMiddleware`](https://reference.langchain.com/python/langchain/agents/middleware/tool_retry/ToolRetryMiddleware) | Retry failed tool calls, optionally scoped to specific tools |

### Data privacy

Detect and handle sensitive data before it reaches the model or gets stored in logs:

```python
from langchain.agents.middleware import PIIMiddleware
from deepagents import create_deep_agent

agent = create_deep_agent(
    model="claude-sonnet-4-6",
    middleware=[
        PIIMiddleware("email", strategy="redact", apply_to_input=True),
        PIIMiddleware("credit_card", strategy="mask", apply_to_input=True),
    ],
)
```




| Middleware | Purpose |
|---|---|
| [`PIIMiddleware`](https://reference.langchain.com/python/langchain/agents/middleware/pii/PIIMiddleware) | Detect and redact or mask PII (emails, credit cards, custom patterns) |

For full configuration options (backoff, jitter, exception filtering, PII strategies, custom detectors), see [prebuilt middleware](/oss/python/langchain/middleware/built-in).
