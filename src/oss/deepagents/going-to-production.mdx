A local prototype typically runs as a single user in a single conversation. Going to production introduces new challenges: multiple users need isolated state, conversations need clear boundaries between ephemeral scratch work and persistent memory, and your agent needs to handle failures and stay within cost and safety limits.

This page covers:
- **[LangSmith Deployments](#langsmith-deployments)**: managed infrastructure with auth, webhooks, and cron
- **[Memory](#memory)**: persist information across conversations
- **[Execution environment](#execution-environment)**: file storage and code execution
- **[Guardrails](#guardrails)**: rate limiting, error handling, and data privacy
- **[Frontend](#frontend)**: connect your UI to a deployed agent

## LangSmith Deployments

The fastest way to get a Deep Agent into production is [LangSmith Deployments](/langsmith/deployment). It provisions the infrastructure your agent needs — [assistants](/langsmith/assistants), [threads](/langsmith/use-threads), [runs](/langsmith/runs), a store, and a checkpointer — so you don't have to set these up yourself. It also gives you [authentication](/langsmith/auth), [webhooks](/langsmith/use-webhooks), [cron jobs](/langsmith/cron-jobs), and [observability](/langsmith/observability) out of the box, and can expose your agent via [MCP](/langsmith/server-mcp) or [A2A](/langsmith/server-a2a).

For setup instructions, see the [LangSmith Deployments quickstart](/langsmith/deployment-quickstart).

## Memory

Without memory, every conversation starts from scratch. Memory lets your agent retain information across conversations — user preferences, learned instructions, past experiences — so it can personalize its behavior over time. For an overview of memory types, see the [memory concepts guide](/oss/concepts/memory).

### Scoping

Memories have different scopes depending on how they're shared:

- **Assistant** (most common): all users of the same assistant share memory
- **Assistant × user**: each user gets private memory within a given assistant
- **Global (user)**: memory that follows a user across all assistants
- **Global (org)**: memory shared across the entire org

For the full scoping API, see [namespace factories](/oss/deepagents/backends#namespace-factories).

### Configuration

In Deep Agents, memory is stored as files in a virtual filesystem. By default, files only last for a single conversation. To persist them, route a path like `/memories/` to a @[`StoreBackend`] that writes to the LangGraph [Store](/langsmith/custom-store). Use a @[`CompositeBackend`] to give the agent both ephemeral scratch space and persistent [long-term memory](/oss/deepagents/long-term-memory).

## Execution environment

Locally, agents can read and write files on disk and run shell commands directly. In production, you need to think about isolation and persistence.

### Filesystem

`FilesystemBackend` and `LocalShellBackend` access the host directly — don't use them in deployed agents. Choose a backend based on what needs to persist:

- @[`StateBackend`]: ephemeral scratch space, scoped to a single conversation. Checkpointed at every step, so avoid writing large files.
- @[`StoreBackend`]: persistent storage that survives across conversations. Scope with a [namespace factory](/oss/deepagents/backends#namespace-factories).
- @[`CompositeBackend`]: mix both — ephemeral scratch by default with persistent routes for specific paths like `/memories/`.

For the full list of backends and how to build custom ones, see [backends](/oss/deepagents/backends).

### Sandboxes

Agents that run shell commands or install packages need an isolated execution environment. [Sandboxes](/oss/deepagents/sandboxes) provide both a filesystem and an `execute` tool for running shell commands inside an isolated container. This isolation also protects your host: if the agent's code exhausts memory or crashes, only the sandbox is affected — your server keeps running.

#### Lifecycle

The key decision is how long a sandbox lives. Does each conversation get a fresh one, or do conversations share a persistent environment?

| Scope | Sandbox ID stored on | Lifecycle | Example use case |
|---|---|---|---|
| **Thread-scoped** | [Thread](/langsmith/use-threads) metadata | Fresh per conversation, cleaned up on TTL | A data analysis bot where each conversation starts clean |
| **Assistant-scoped** | [Assistant](/langsmith/assistants) config | Shared across all conversations | A coding assistant that maintains a cloned repo across conversations |

    Each conversation gets its own sandbox. The agent reads `thread_id` from the runtime config via `get_config()`, so each [thread](/langsmith/use-threads) automatically gets its own isolated environment. Cleaned up when the sandbox [TTL](/langsmith/configure-ttl) expires.

:::python
```python agent.py
from daytona import CreateSandboxFromSnapshotParams, Daytona
from deepagents import create_deep_agent
from langchain_daytona import DaytonaSandbox
from langgraph.config import get_config

client = Daytona()


def make_backend(rt):
    thread_id = get_config()["configurable"]["thread_id"]
    try:
        sandbox = client.find_one(labels={"thread_id": thread_id})
    except Exception:
        sandbox = client.create(
            CreateSandboxFromSnapshotParams(
                labels={"thread_id": thread_id},
                auto_delete_interval=3600,  # TTL: clean up when idle
            )
        )
    return DaytonaSandbox(sandbox=sandbox)


agent = create_deep_agent(backend=make_backend)
```
:::

:::js

```typescript src/agent.ts
import { createDeepAgent } from "deepagents";

const client = new Daytona();

const agent = createDeepAgent({
  backend: async (rt) => {
    const { thread_id: threadId } = getConfig().configurable;
    let sandbox;
    try {
      sandbox = await client.findOne({ labels: { thread_id: threadId } });
    } catch {
      sandbox = await client.create({
        labels: { thread_id: threadId },
        autoDeleteInterval: 3600, // TTL: clean up when idle
      });
    }
    return DaytonaSandbox.fromId(sandbox.id);
  },
});

```

#### File transfers

Sandboxes are isolated containers — your application code can't directly access files inside them. Use `upload_files()` and `download_files()` to move data across the sandbox boundary:

- **Seed the sandbox before the agent runs** — upload user files, [skill](/oss/deepagents/skills) scripts, configuration, or [persistent memories](/oss/deepagents/long-term-memory) so the agent has what it needs from the start
- **Retrieve results after the agent finishes** — download generated artifacts (reports, plots, exports) and sync updated memories back for future conversations

For provider-specific file transfer examples, see [working with files](/oss/deepagents/sandboxes#working-with-files). For provider setup, security, and lifecycle patterns, see the full [sandboxes guide](/oss/deepagents/sandboxes).

<Accordion title="Example: syncing skills and memories with custom middleware">
  [Skills](/oss/deepagents/skills) and [memories](/oss/deepagents/long-term-memory) stored in the [Store](/langsmith/custom-store) need to be physically present in the sandbox for `execute()`. Use [custom middleware](/oss/langchain/middleware/custom) with `before_agent` and `after_agent` hooks to sync them across the boundary:

:::python
```python agent.py
from deepagents import create_deep_agent
from langchain.agents.middleware import AgentMiddleware, AgentState
from langgraph.runtime import Runtime


class SandboxSyncMiddleware(AgentMiddleware):
    """Sync skills and memories between the Store and the sandbox."""

    def __init__(
        self,
        backend,
        skills_namespace: tuple[str, ...],
        memories_namespace: tuple[str, ...],
    ):
        super().__init__()
        self.backend = backend
        self.skills_namespace = skills_namespace
        self.memories_namespace = memories_namespace

    def before_agent(self, state: AgentState, runtime: Runtime) -> None:
        """Upload skills and memories into the sandbox."""
        store = runtime.store
        files = []
        for item in store.search(self.skills_namespace):
            files.append((f"/skills/{item.key}", item.value["content"].encode()))
        for item in store.search(self.memories_namespace):
            files.append((f"/memories/{item.key}", item.value["content"].encode()))
        if files:
            self.backend.upload_files(files)

    def after_agent(self, state: AgentState, runtime: Runtime) -> None:
        """Sync updated memories back to the Store."""
        store = runtime.store
        results = self.backend.download_files(
            [f"/memories/{item.key}" for item in store.search(self.memories_namespace)]
        )
        for result in results:
            if result.content is not None:
                store.put(
                    self.memories_namespace,
                    result.path.split("/")[-1],
                    {"content": result.content.decode()},
                )


backend = CompositeBackend(
    default=DaytonaSandbox(sandbox=sandbox),
    routes={
        "/memories/": StoreBackend(
            rt,
            namespace=lambda ctx: (ctx.runtime.context.user_id,),
        ),
    },
)

agent = create_deep_agent(
    backend=backend,
    middleware=[
        SandboxSyncMiddleware(
            backend=backend,
            skills_namespace=("user_123", "skills"),
            memories_namespace=("user_123", "memories"),
        ),
    ],
)
```
:::

:::js
```typescript src/agent.ts
import { createMiddleware } from "langchain";
import { createDeepAgent, CompositeBackend, StoreBackend } from "deepagents";

const createSandboxSyncMiddleware = (
  backend: CompositeBackend,
  skillsNamespace: string[],
  memoriesNamespace: string[],
) => {
  return createMiddleware({
    name: "SandboxSyncMiddleware",
    beforeAgent: async (state, runtime) => {
      // Upload skills and memories into the sandbox
      const store = runtime.store;
      const encoder = new TextEncoder();
      const files: [string, Uint8Array][] = [];
      for (const item of await store.search(skillsNamespace)) {
        files.push([`/skills/${item.key}`, encoder.encode(item.value.content)]);
      }
      for (const item of await store.search(memoriesNamespace)) {
        files.push([`/memories/${item.key}`, encoder.encode(item.value.content)]);
      }
      if (files.length > 0) {
        await backend.uploadFiles(files);
      }
    },
    afterAgent: async (state, runtime) => {
      // Sync updated memories back to the Store
      const store = runtime.store;
      const items = await store.search(memoriesNamespace);
      const results = await backend.downloadFiles(
        items.map((item) => `/memories/${item.key}`),
      );
      const decoder = new TextDecoder();
      for (const result of results) {
        if (result.content) {
          await store.put(
            memoriesNamespace,
            result.path.split("/").pop()!,
            { content: decoder.decode(result.content) },
          );
        }
      }
    },
  });
};

const backend = new CompositeBackend(
  await DaytonaSandbox.fromId(sandbox.id),
  {
    "/memories/": new StoreBackend(rt, {
      namespace: (ctx) => [ctx.runtime.context.userId],
    }),
  },
);

const agent = createDeepAgent({
  backend,
  middleware: [
    createSandboxSyncMiddleware(
      backend,
      ["user_123", "skills"],
      ["user_123", "memories"],
    ),
  ],
});

export { agent };
```

## Guardrails

Agents in production run autonomously, which means they can loop indefinitely, hit rate limits, or process user data that contains sensitive information. Deep Agents support [middleware](/oss/langchain/middleware/built-in) that wraps model and tool calls to handle these concerns.

### Rate limiting

Without limits, a confused agent can burn through your API budget in minutes by looping on the same tool call or making hundreds of model calls. Set caps on both model calls and tool executions per run:

:::python
```python
from deepagents import create_deep_agent
from langchain.agents.middleware import ModelCallLimitMiddleware, ToolCallLimitMiddleware

agent = create_deep_agent(
    model="claude-sonnet-4-6",
    middleware=[
        ModelCallLimitMiddleware(run_limit=50),
        ToolCallLimitMiddleware(run_limit=200),
    ],
)
```
:::

:::js
```typescript
import { createAgent, modelCallLimitMiddleware, toolCallLimitMiddleware } from "langchain";

const agent = createAgent({
  model: "claude-sonnet-4-6",
  middleware: [
    modelCallLimitMiddleware({ runLimit: 50 }),
    toolCallLimitMiddleware({ runLimit: 200 }),
  ],
});
```
:::

Use `run_limit` to cap calls within a single invocation (resets each turn). Use `thread_limit` to cap calls across an entire conversation (requires a checkpointer). See @[`ModelCallLimitMiddleware`] and @[`ToolCallLimitMiddleware`] for the full configuration.

### Handling errors

Not all errors should be handled the same way. Transient failures (network timeouts, rate limits) should be retried automatically. Errors the LLM can recover from (bad tool output, parsing failures) should be fed back to the model. Errors that need human input should pause the agent. For the full breakdown with code examples, see [Handle errors appropriately](/oss/langgraph/thinking-in-langgraph#handle-errors-appropriately).

Middleware handles the transient case. Model calls and tool calls each have their own retry middleware with exponential backoff. If your primary model provider goes down entirely, the fallback middleware switches to an alternative:

:::python
```python
from deepagents import create_deep_agent
from langchain.agents.middleware import (
    ModelFallbackMiddleware,
    ModelRetryMiddleware,
    ToolRetryMiddleware,
)

agent = create_deep_agent(
    model="claude-sonnet-4-6",
    middleware=[
        # Retry model calls on rate limits, timeouts, and 5xx errors
        ModelRetryMiddleware(max_retries=3, backoff_factor=2.0, initial_delay=1.0),
        # If the primary model is fully down, fall back to an alternative
        ModelFallbackMiddleware("gpt-4.1"),
        # Retry specific tools that hit external APIs (not all tools)
        ToolRetryMiddleware(max_retries=2, tools=["search", "fetch_url"]),
    ],
)
```
:::

:::js
```typescript
import {
  createAgent,
  modelFallbackMiddleware,
  modelRetryMiddleware,
  toolRetryMiddleware,
} from "langchain";

const agent = createAgent({
  model: "claude-sonnet-4-6",
  middleware: [
    // Retry model calls on rate limits, timeouts, and 5xx errors
    modelRetryMiddleware({ maxRetries: 3, backoffFactor: 2.0, initialDelayMs: 1000 }),
    // If the primary model is fully down, fall back to an alternative
    modelFallbackMiddleware("gpt-4.1"),
    // Retry specific tools that hit external APIs (not all tools)
    toolRetryMiddleware({ maxRetries: 2, tools: ["search", "fetch_url"] }),
  ],
});
```
:::

Scope @[`ToolRetryMiddleware`] to specific tools rather than retrying everything — a filesystem `read_file` that fails won't benefit from a retry, but a web search that times out probably will. See @[`ModelRetryMiddleware`] and @[`ModelFallbackMiddleware`] for the full configuration.

### Data privacy

If your agent processes user input that might contain emails, credit card numbers, or other PII, you can detect and handle it before it reaches the model or gets stored in logs:

:::python
```python
from deepagents import create_deep_agent
from langchain.agents.middleware import PIIMiddleware

agent = create_deep_agent(
    model="claude-sonnet-4-6",
    middleware=[
        PIIMiddleware("email", strategy="redact", apply_to_input=True),
        PIIMiddleware("credit_card", strategy="mask", apply_to_input=True),
    ],
)
```
:::

:::js
```typescript
import { createAgent, piiMiddleware } from "langchain";

const agent = createAgent({
  model: "claude-sonnet-4-6",
  middleware: [
    piiMiddleware("email", { strategy: "redact", applyToInput: true }),
    piiMiddleware("credit_card", { strategy: "mask", applyToInput: true }),
  ],
});
```
:::

Strategies include `redact` (replace with `[REDACTED_EMAIL]`), `mask` (partial masking like `****-****-****-1234`), `hash` (deterministic hash), and `block` (raise an error). You can also write custom detectors for domain-specific patterns. See @[`PIIMiddleware`] for the full configuration.

For the complete list of available middleware, see [prebuilt middleware](/oss/langchain/middleware/built-in).

### Custom middleware

You can hook into any point in the agent loop to add your own business logic. Available hooks include `before_model`, `after_model`, `wrap_model_call`, and `wrap_tool_call`. See [custom middleware](/oss/langchain/middleware/custom) for details.

## Frontend

Deep Agents use [`useStream`](/oss/langchain/frontend/overview) to connect your UI to the agent backend. `useStream` is a frontend hook (available for React, Vue, Svelte, and Angular) that streams messages, subagent progress, and custom state from your agent in real time.

Locally, `useStream` points at `http://localhost:2024`. In production, point it at your [LangSmith Deployment](/langsmith/deployment) and configure reconnection so users don't lose progress if their connection drops.

```tsx
import { useStream } from "@langchain/react";

function App() {
  const stream = useStream<typeof agent>({
    apiUrl: "https://your-deployment.langsmith.dev",
    assistantId: "agent",
    reconnectOnMount: true,    // Resume stream after page refresh or navigation
    fetchStateHistory: true,   // Load full thread history on mount
  });
}
```

`reconnectOnMount` picks up an in-progress run automatically — if a user refreshes while the agent is working, they'll see it continue rather than a blank screen. `fetchStateHistory` loads the full conversation history for the thread, so returning users see previous messages.

For deep agent workflows that spawn many subagents, set a high `recursionLimit` when submitting to avoid cutting off long-running executions:

```tsx
stream.submit(
  { messages: [{ type: "human", content: text }] },
  {
    streamSubgraphs: true,
    config: { recursionLimit: 10000 },
  },
);
```

For UI patterns specific to deep agents — subagent cards, todo lists, and custom state rendering — see the [frontend guide](/oss/deepagents/frontend/overview).
