This page covers how to take your local Deep Agent to production.

Deep Agents deploy on [LangSmith Deployment](/langsmith/deployment) (LSD), which handles infrastructure for you — [assistants](/langsmith/assistants), [threads](/langsmith/use-threads), [runs](/langsmith/runs), a provisioned store and checkpointer — and can expose your agent via [MCP](/langsmith/server-mcp), [A2A](/langsmith/server-a2a), [webhooks](/langsmith/use-webhooks), or [cron](/langsmith/cron-jobs). For general deployment setup, see the [LangSmith Deployment quickstart](/langsmith/deployment-quickstart).

## Overview

When deploying a Deep Agent, there are a few key decisions to make:

- **[Memory](#memory)** — How to scope persistent data (preferences, instructions, [skills](/oss/deepagents/skills)) that carries across conversations
- **[Files](#files)** — How to configure the agent's filesystem — which [backend](/oss/deepagents/backends) to use, what's ephemeral vs persistent
- **[Sandboxes](#sandboxes)** — How to manage isolated environments for code execution
- **[Resilience middleware](#production-middleware)** — How to handle transient failures like rate limits and provider outages

## Memory

A deployed agent needs to remember things across conversations — user preferences, learned instructions, project context. Without persistent memory, every conversation starts from scratch. This persistent storage is also where [skills](/oss/deepagents/skills) (SKILL.md) and agent memories (AGENTS.md) are loaded from.

Use a @[`CompositeBackend`] to give the agent both ephemeral scratch space and persistent [long-term memory](/oss/deepagents/long-term-memory). Anything the agent writes to `/memories/` persists in the LangGraph [Store](/langsmith/custom-store) that LSD provisions automatically. Everything else is ephemeral and scoped to the current conversation.

The [namespace factory](/oss/deepagents/backends#namespace-factories) controls who can access that persistent memory — whether it's private to a single user or shared across an [assistant](/langsmith/assistants):

<Tabs>
  <Tab title="Per-user (most common)">
    Each user gets private persistent memory across all their conversations. Use this for personal preferences, instructions, and learned context.

:::python
```python
from deepagents import create_deep_agent
from deepagents.backends import CompositeBackend, StateBackend, StoreBackend

agent = create_deep_agent(
    backend=lambda rt: CompositeBackend(
        default=StateBackend(rt),  # Ephemeral scratch space
        routes={
            "/memories/": StoreBackend(
                rt,
                namespace=lambda ctx: (ctx.runtime.context.user_id,),
            ),
        },
    ),
)
```
:::

:::js
```typescript
import { createDeepAgent, CompositeBackend, StateBackend, StoreBackend } from "deepagents";

const agent = createDeepAgent({
  backend: (rt) => new CompositeBackend(
    new StateBackend(rt),  // Ephemeral scratch space
    {
      "/memories/": new StoreBackend(rt, {
        namespace: (ctx) => [ctx.runtime.context.userId],
      }),
    },
  ),
});
```
:::

  </Tab>
  <Tab title="Per-assistant">
    All users of the same [assistant](/langsmith/assistants) share memory. Use this for shared team knowledge or project documentation.

:::python
```python
from langgraph.config import get_config
from deepagents import create_deep_agent
from deepagents.backends import CompositeBackend, StateBackend, StoreBackend

agent = create_deep_agent(
    backend=lambda rt: CompositeBackend(
        default=StateBackend(rt),
        routes={
            "/memories/": StoreBackend(
                rt,
                namespace=lambda ctx: (
                    get_config()["metadata"]["assistant_id"],
                ),
            ),
        },
    ),
)
```
:::

:::js
```typescript
import { getConfig } from "@langchain/langgraph";
import { createDeepAgent, CompositeBackend, StateBackend, StoreBackend } from "deepagents";

const agent = createDeepAgent({
  backend: (rt) => new CompositeBackend(
    new StateBackend(rt),
    {
      "/memories/": new StoreBackend(rt, {
        namespace: (ctx) => {
          const config = getConfig();
          return [config.metadata.assistantId];
        },
      }),
    },
  ),
});
```
:::

  </Tab>
  <Tab title="Per-user per-thread">
    Memory is private to a specific user *and* a specific conversation. Use this when the agent's persistent state should be conversation-specific — for example, a project workspace that each thread manages independently.

:::python
```python
from langgraph.config import get_config
from deepagents import create_deep_agent
from deepagents.backends import CompositeBackend, StateBackend, StoreBackend

agent = create_deep_agent(
    backend=lambda rt: CompositeBackend(
        default=StateBackend(rt),
        routes={
            "/memories/": StoreBackend(
                rt,
                namespace=lambda ctx: (
                    ctx.runtime.context.user_id,
                    get_config()["configurable"]["thread_id"],
                ),
            ),
        },
    ),
)
```
:::

:::js
```typescript
import { getConfig } from "@langchain/langgraph";
import { createDeepAgent, CompositeBackend, StateBackend, StoreBackend } from "deepagents";

const agent = createDeepAgent({
  backend: (rt) => new CompositeBackend(
    new StateBackend(rt),
    {
      "/memories/": new StoreBackend(rt, {
        namespace: (ctx) => {
          const config = getConfig();
          return [ctx.runtime.context.userId, config.configurable.threadId];
        },
      }),
    },
  ),
});
```
:::

  </Tab>
</Tabs>

You can also read and write to the store from your application code using the [Store API](/langsmith/custom-store). See [accessing memories from external code](/oss/deepagents/long-term-memory#accessing-memories-from-external-code) for examples.

For the full namespace factory API, see [namespace factories](/oss/deepagents/backends#namespace-factories). For memory patterns like self-improving instructions and knowledge bases, see [long-term memory](/oss/deepagents/long-term-memory).

## Files

Deep Agents expose a virtual filesystem to the agent via tools like `read_file`, `write_file`, `edit_file`, `ls`, `glob`, and `grep`. The [backend](/oss/deepagents/backends) you choose determines where those files live and how long they last.

| Backend | Files persist | Scope | Best for |
|---|---|---|---|
| @[`StateBackend`] (default) | Within a single conversation | Per-[thread](/langsmith/use-threads) | Ephemeral scratch space — drafts, intermediate results |
| @[`StoreBackend`] | Across conversations | Configurable via [namespace factory](/oss/deepagents/backends#namespace-factories) | Persistent working files, shared project state |
| [Sandbox](/oss/deepagents/sandboxes) | Depends on [scope](#sandboxes) | Per-thread or per-assistant | Code execution with shell access |

Most deployed agents use a @[`CompositeBackend`] to combine these — for example, `StateBackend` or a sandbox for working files plus `StoreBackend` for persistent `/memories/`. This is the pattern shown in the [memory examples above](#memory).

<Warning>
    Do not use `FilesystemBackend` in deployed agents. It accesses the host filesystem directly and is not isolated between users or threads.
</Warning>

For the full list of backends and how to build custom ones, see [backends](/oss/deepagents/backends).

## Sandboxes

If your agent needs to execute code, it needs an isolated environment. [Sandboxes](/oss/deepagents/sandboxes) give the agent a filesystem and shell access in a container that can't affect your infrastructure. In production, always use a sandbox instead of `LocalShellBackend`, which has no isolation.

The key decision is how long a sandbox lives — does each conversation get a fresh one, or do conversations share a persistent environment?

| Scope | Sandbox ID stored on | Lifecycle | Best for |
|---|---|---|---|
| **Thread-scoped** | [Thread](/langsmith/use-threads) metadata | Fresh per conversation, cleaned up on TTL | Data analysis, one-off coding |
| **Assistant-scoped** | [Assistant](/langsmith/assistants) config | Shared across all conversations | Persistent dev environments |

<Tabs>
  <Tab title="Thread-scoped (most common)">
    Each conversation gets its own sandbox. The sandbox is created when the first [run](/langsmith/runs) starts and reused for follow-up messages on the same [thread](/langsmith/use-threads). Cleaned up when the thread ends or the sandbox [TTL](/langsmith/configure-ttl) expires. Store the sandbox ID in thread metadata so follow-up runs reconnect.
  </Tab>
  <Tab title="Assistant-scoped">
    All conversations share one sandbox. The sandbox ID is stored on the [assistant](/langsmith/assistants) config, so every conversation returns to the same environment. Files, installed packages, and cloned repositories persist across conversations.

    <Warning>
        Assistant-scoped sandboxes accumulate state over time. Implement periodic cleanup, use snapshots to reset, or configure a TTL to prevent unbounded resource usage.
    </Warning>
  </Tab>
</Tabs>

To combine a sandbox with persistent memory, use the sandbox as the default backend in a `CompositeBackend`:

:::python
```python
from deepagents import create_deep_agent
from deepagents.backends import CompositeBackend, StoreBackend

agent = create_deep_agent(
    backend=lambda rt: CompositeBackend(
        default=sandbox_backend,  # Code execution + working files
        routes={
            "/memories/": StoreBackend(
                rt,
                namespace=lambda ctx: (ctx.runtime.context.user_id,),
            ),
        },
    ),
)
```
:::

:::js
```typescript
import { createDeepAgent, CompositeBackend, StoreBackend } from "deepagents";

const agent = createDeepAgent({
  backend: (rt) => new CompositeBackend(
    sandboxBackend,  // Code execution + working files
    {
      "/memories/": new StoreBackend(rt, {
        namespace: (ctx) => [ctx.runtime.context.userId],
      }),
    },
  ),
});
```
:::

For provider setup, security, and lifecycle patterns, see the full [sandboxes guide](/oss/deepagents/sandboxes).

## Production middleware

In production, transient failures are common — model API rate limits, provider outages, flaky tool calls. Add [resilience middleware](/oss/langchain/middleware/built-in) to handle these automatically:

:::python
```python
from langchain.agents.middleware import (
    ModelRetryMiddleware,
    ModelFallbackMiddleware,
    ToolRetryMiddleware,
)
from deepagents import create_deep_agent

agent = create_deep_agent(
    model="claude-sonnet-4-6",
    middleware=[
        ModelRetryMiddleware(max_retries=3),  # Retry on transient API failures
        ModelFallbackMiddleware("gpt-4.1"),   # Fall back if primary model is down
        ToolRetryMiddleware(max_retries=2),   # Retry flaky tool calls
    ],
    ...
)
```
:::

:::js
```typescript
import { createDeepAgent } from "deepagents";
import {
  ModelRetryMiddleware,
  ModelFallbackMiddleware,
  ToolRetryMiddleware,
} from "@langchain/langgraph";

const agent = createDeepAgent({
  model: "claude-sonnet-4-6",
  middleware: [
    new ModelRetryMiddleware({ maxRetries: 3 }),  // Retry on transient API failures
    new ModelFallbackMiddleware("gpt-4.1"),       // Fall back if primary model is down
    new ToolRetryMiddleware({ maxRetries: 2 }),   // Retry flaky tool calls
  ],
  ...
});
```
:::

| Middleware | Purpose |
|---|---|
| @[`ModelRetryMiddleware`] | Retry on rate limits, timeouts, server errors |
| @[`ModelFallbackMiddleware`] | Try alternative models on failure |
| @[`ToolRetryMiddleware`] | Retry flaky tool calls |
| @[`ModelCallLimitMiddleware`] | Cap model calls per run |
| @[`ToolCallLimitMiddleware`] | Cap tool executions per run |

For full configuration options (backoff, jitter, exception filtering, failure handling), see [prebuilt middleware](/oss/langchain/middleware/built-in).
